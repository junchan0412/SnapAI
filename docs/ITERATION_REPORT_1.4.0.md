# SnapAI 1.4.0 迭代报告

## 目标

1.4.0 延续 1.3.0 阶段报告中的后续方向,重点推进:

- 本地模型和隐私模式深化
- 写回兼容性矩阵和自动化回归
- 可组合动作工作流
- 用户可观察性和失败诊断
- 更新器职责收敛与发布可信度

## 当前完成

### 1. 隐私模式本地模型优先

- 新增 LM Studio 本地供应商预设,默认使用 `http://localhost:1234/v1`。
- 为 AI 供应商增加本地端点识别,覆盖 `localhost`、`127.0.0.1` 和 `::1`。
- 新增本地模型健康诊断,可识别 Ollama、LM Studio 和通用本地 OpenAI 兼容服务。
- 当本地供应商缺少占位 API Key 或没有启用模型时,诊断会提示如何启动本地服务、拉取/加载模型和填写占位 Key。
- 隐私模式下如果存在可请求的本地模型,自动路由会优先选择本地端点,并在候选原因中标记为“本地隐私优先”。
- 隐私模式保留云端候选时会标记为“云端备用模型”,便于用户从诊断中识别本地失败后的 fallback 方向。
- 手动关闭自动路由时仍尊重用户当前选择的模型,避免隐私模式覆盖明确的手动操作。
- 补充逻辑测试覆盖 Ollama/LM Studio 本地识别、隐私模式本地优先、本地健康恢复建议和手动路由保留行为。

### 2. 写回兼容性矩阵

- 新增写回兼容性矩阵,首批覆盖 Safari、Chrome、Edge、微信、飞书、Obsidian、Notion、Xcode 和 Word。
- 写回失败或降级为复制时,诊断建议会结合目标应用给出更具体的恢复路径。
- 兼容性建议默认只输出静态指导,不记录用户文本内容。
- 补充逻辑测试覆盖应用别名匹配、未知应用降级和浏览器写回失败建议。

### 3. 可组合动作工作流诊断

- 新增动作 pipeline 诊断结构,将动作执行拆解为输入、隐私、输出和模型策略。
- 请求诊断会记录 pipeline 元数据,例如 `text+image`、`preview+local-redaction`、`replace-confirmation` 和 `auto-route-local-first`。
- 当前实现先采用派生型元数据,不改变用户已有动作配置结构,为后续可视化 pipeline 编辑器打基础。
- 补充逻辑测试覆盖 pipeline 派生规则和请求诊断输出。

### 4. 隐私模式云端 fallback 审核

- 在请求诊断中新增云端 fallback 审核摘要,显示本地候选和云端候选数量。
- 隐私模式本地优先下,如果本地模型失败且下一个候选是云端模型,不会静默自动切换。
- 诊断会给出 `fallback-cloud-confirmation-required` 恢复代码,提示用户手动选择云端模型或关闭严格本地优先后重试。
- 补充逻辑测试覆盖 fallback 决策、诊断摘要和本地到云端的确认策略。

### 5. 用户可观察性和健康中心

- 权限健康中心新增最近一次 AI 请求状态。
- 复制完整或精简诊断时会包含最近请求 outcome、恢复代码、恢复建议和最近尝试摘要。
- 最近请求失败会进入修复建议列表,并继续通过脱敏逻辑保护路径、密钥和敏感错误片段。
- 安装日志、取词状态、写回状态、签名、快捷键、供应商可请求性和最近 AI 请求现在能在同一个健康快照中排查。

## 发布收尾

- 更新 `Resources/Info.plist` 到 `1.4.0`。
- 新增 `docs/RELEASE_NOTES_1.4.0.md`。
- 更新 README 首屏重点和 UI 总览图。
- 发布前执行 `scripts/preflight-release.sh`,生成 `SnapAI-v1.4.0.zip` 和 `snapai-manifest-v1.4.0.json`。

## 后续方向

- 将动作 pipeline 从诊断元数据升级为可视化配置器。
- 为本地模型增加可选联网健康探测,例如检测 Ollama/LM Studio 当前加载模型。
- 为云端 fallback 增加结果面板内的一键确认继续能力。
- 将更新器进一步收敛到独立 helper,减少安装脚本职责。
