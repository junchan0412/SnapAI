# SnapAI 1.3.0 阶段性迭代报告

## 阶段判断

1.3.0 的核心价值不是单点功能增加,而是把 SnapAI 的基础体验补成一个更稳定的闭环:用户能更快触发动作,更清楚知道数据会发往哪里,更容易恢复失败操作,也更容易在无 Apple Developer 账号的开源分发条件下完成安装和更新。

## 已完成方向

### 1. 操作入口从菜单栏扩展到工作台

- `Command + K` 命令面板成为高频入口,覆盖动作、模型、上下文、历史和设置项。
- 正式应用菜单与快捷键体系更完整,减少用户在菜单栏层级里寻找功能的成本。
- URL Scheme 和 macOS Services 提供自动化入口,便于接入 Shortcuts、Raycast、Alfred 和脚本。

### 2. 结果面板更接近可恢复的工作流

- 替换、追加、复制、继续追问、重新生成等行为被明确为结果面板操作。
- 写回前引入 Diff 预览和事务保护,降低误替换风险。
- 替换/追加时保护用户剪贴板,并尽量恢复触发前的焦点应用。

### 3. 历史记录从最近列表升级为知识库

- 独立历史窗口支持搜索、收藏、删除单条和多维筛选。
- 历史可以导出 Markdown,也可以转化为上下文包,服务后续请求。
- 隐私标签和仅元信息模式让历史更适合长期保留。

### 4. 隐私与诊断成为产品能力

- 发送前预览、本地脱敏、高风险内容强制确认形成更透明的发送链路。
- 默认脱敏覆盖邮箱、手机号、API Key、访问令牌、URL 参数密钥、JWT 和私钥块。
- 旧版默认脱敏规则会在本地升级或导入旧配置时迁移到当前规则集。
- 复制诊断时会脱敏路径、Key、token 和常见错误片段。

### 5. 更新和签名策略适配开源分发现实

- 当前没有 Apple Developer ID,所以继续采用稳定本地签名、manifest、SHA256、bundle id 和 designated requirement 校验。
- 应用内更新尽量自动处理 quarantine,并在替换后尝试重新启动。
- README 明确解释 `xattr -cr`、辅助功能授权、Keychain 重授权和稳定签名身份之间的关系。

## 当前仍需关注的风险

- 未公证应用首次安装仍会触发 Gatekeeper/quarantine 相关门槛,只能通过清晰文档和应用内更新减轻。
- 不同 macOS 应用对 Accessibility、Services、模拟粘贴的支持差异很大,后续需要继续积累兼容性矩阵。
- 自签名证书如果在发布机上丢失或变更,仍可能导致辅助功能和 Keychain 权限重新授权。
- AI 路由已经具备基础判断,但还需要真实使用数据来优化成本、速度和质量的默认权重。

## 后续迭代建议

### 1. 更新器长期独立化

将更新安装逻辑逐步迁移到独立 helper updater,减少 shell 脚本承担的职责。目标是更可靠地处理退出、替换、重启、失败回滚和安装日志。

### 2. 本地模型和隐私模式深化

完善 Ollama、LM Studio、本地 OpenAI-compatible endpoint 的体验。对敏感动作支持“仅本地模型”,并在云端模型 fallback 前明确提示用户。

### 3. 兼容性矩阵和自动化回归

建立常见应用写回矩阵,优先覆盖 Safari、Chrome、微信、飞书、Obsidian、Notion、Xcode、Word。用自动化测试继续覆盖版本比较、Release tag 解析、Base URL 规范化、热键冲突、Prompt 渲染、模型 fallback 和写回事务。

### 4. 可组合动作工作流

把动作升级为 pipeline:输入来源、脱敏、模型路由、后处理、输出位置可以组合。这样 SnapAI 可以从单次 AI 调用扩展为可复用工作流。

### 5. 用户可观察性

继续增强权限健康中心和请求诊断,提供最近一次失败的路由、网络、权限、写回和更新状态摘要,但始终默认脱敏。
